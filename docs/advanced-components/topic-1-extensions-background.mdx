## Why extensions are different

An app extension is not a module inside your app — it's a separate target that
ships inside your `.app` bundle, is launched by iOS (not by you), and runs in
its own sandbox with its own memory budget (often ~60–120 MB).

<Callout type="warning" title="Extensions are a different binary and a different process">
  You cannot call into your app's code directly from an extension. You share
  files via an <strong>App Group</strong>, KV data via <code>UserDefaults(suiteName:)</code>,
  and secrets via a Keychain access group. Any singleton you rely on in the app
  does not exist in the extension. Design the extension's data layer from
  scratch, and treat the app as a peer, not a parent.
</Callout>

## App Groups — the shared sandbox

Enable App Groups on *both* targets (app + extension) in Signing & Capabilities,
pick an identifier like `group.com.example.shop`, and you get a shared
container URL you can read and write from either process.

```swift
enum SharedStore {
    static let groupID = "group.com.example.shop"

    static var containerURL: URL {
        FileManager.default
            .containerURL(forSecurityApplicationGroupIdentifier: groupID)!
    }

    static var defaults: UserDefaults {
        UserDefaults(suiteName: groupID)!
    }

    static func writeDraft(_ draft: Draft) throws {
        let url = containerURL.appendingPathComponent("draft.json")
        try JSONEncoder().encode(draft).write(to: url, options: .atomic)
    }
}
```

Use file coordination (`NSFileCoordinator`) if two processes can write the
same file simultaneously — otherwise you will corrupt data.

## Notification Service Extension — rich push

Rich push (image, video, decrypted body) requires a Notification Service
Extension. Your server sends `"mutable-content": 1` and a URL; the extension
downloads and attaches:

```swift
final class NotificationService: UNNotificationServiceExtension {
    var handler: ((UNNotificationContent) -> Void)?
    var best: UNMutableNotificationContent?

    override func didReceive(_ request: UNNotificationRequest,
                             withContentHandler ch: @escaping (UNNotificationContent) -> Void) {
        handler = ch
        best = request.content.mutableCopy() as? UNMutableNotificationContent

        guard let urlString = request.content.userInfo["image_url"] as? String,
              let url = URL(string: urlString) else {
            ch(best ?? request.content); return
        }

        URLSession.shared.downloadTask(with: url) { [weak self] file, _, _ in
            guard let self, let file, let best = self.best else { return }
            let dest = FileManager.default.temporaryDirectory
                .appendingPathComponent(UUID().uuidString + ".jpg")
            try? FileManager.default.moveItem(at: file, to: dest)
            if let attach = try? UNNotificationAttachment(identifier: "image", url: dest) {
                best.attachments = [attach]
            }
            ch(best)
        }.resume()
    }

    override func serviceExtensionTimeWillExpire() {
        if let best { handler?(best) }
    }
}
```

You have ~30 seconds. Always call the handler — even on failure — or the user
sees nothing.

## App Intents — one protocol, many surfaces

App Intents (iOS 16+) is the modern replacement for SiriKit for most verbs.
One `AppIntent` struct powers Siri, Shortcuts, Spotlight, and the Action Button.

```swift
struct LogExpenseIntent: AppIntent {
    static var title: LocalizedStringResource = "Log expense"
    static var description = IntentDescription("Record a new expense in Shop.")

    @Parameter(title: "Amount") var amount: Double
    @Parameter(title: "Category") var category: String

    func perform() async throws -> some IntentResult {
        try await ExpenseStore.shared.add(amount: amount, category: category)
        return .result(dialog: "Logged \(amount) to \(category).")
    }
}
```

No separate target needed for basic intents — they can live in the main app.
Put them in a framework if you also want widget configuration to use them.

## BGTaskScheduler — the right way

Two flavors: `BGAppRefreshTask` for short fetches, `BGProcessingTask` for
longer work that runs when the device is idle and charging.

### Complete example

```swift
import BackgroundTasks

enum BG {
    static let refreshID = "com.example.shop.refresh"
    static let cleanupID = "com.example.shop.cleanup"
}

@main
struct ShopApp: App {
    init() {
        BGTaskScheduler.shared.register(forTaskWithIdentifier: BG.refreshID, using: nil) { task in
            handleRefresh(task: task as! BGAppRefreshTask)
        }
        BGTaskScheduler.shared.register(forTaskWithIdentifier: BG.cleanupID, using: nil) { task in
            handleCleanup(task: task as! BGProcessingTask)
        }
    }
    var body: some Scene { WindowGroup { RootView() } }
}

func scheduleRefresh() {
    let req = BGAppRefreshTaskRequest(identifier: BG.refreshID)
    req.earliestBeginDate = .now.addingTimeInterval(15 * 60)
    try? BGTaskScheduler.shared.submit(req)
}

func handleRefresh(task: BGAppRefreshTask) {
    scheduleRefresh() // chain the next one immediately

    let op = Task {
        do {
            try await FeedService.shared.refresh()
            task.setTaskCompleted(success: true)
        } catch {
            task.setTaskCompleted(success: false)
        }
    }
    task.expirationHandler = { op.cancel() }
}

func handleCleanup(task: BGProcessingTask) {
    let req = BGProcessingTaskRequest(identifier: BG.cleanupID)
    req.requiresExternalPower = true
    req.requiresNetworkConnectivity = false
    try? BGTaskScheduler.shared.submit(req)

    let op = Task {
        await Database.shared.vacuum()
        task.setTaskCompleted(success: true)
    }
    task.expirationHandler = { op.cancel() }
}
```

<Callout type="tip" title="Force a task to run in the debugger">
  Set a breakpoint right after you submit a request, then run this in lldb:
  <br />
  <code>e -l objc -- (void)[[BGTaskScheduler sharedScheduler] _simulateLaunchForTaskWithIdentifier:@"com.example.shop.refresh"]</code>
  <br />
  You cannot wait for the real scheduler during development — iOS may take
  hours or days to grant you a slot.
</Callout>

## Background URLSession — uploads and downloads that survive termination

For large transfers that must finish even if the user swipes the app away:

```swift
let config = URLSessionConfiguration.background(withIdentifier: "com.example.shop.bg")
config.isDiscretionary = true           // let iOS pick the best time
config.sessionSendsLaunchEvents = true  // wake the app on completion
let session = URLSession(configuration: config, delegate: self, delegateQueue: nil)
```

Implement `application(_:handleEventsForBackgroundURLSession:completionHandler:)`
in your AppDelegate to resume and finish the work after a cold launch.

## Silent push — server-initiated refresh

Send a push with `"content-available": 1` and no alert/sound/badge. iOS wakes
your app briefly (often) to call `application(_:didReceiveRemoteNotification:)`.
Same budget as a refresh task, same rules: do small, idempotent work, then
finish. Silent push is throttled aggressively — do not rely on it for more
than a few updates per hour.
