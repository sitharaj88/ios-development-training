## Why DI before frameworks

The best dependency injection is the one you don't notice — initializer
parameters. Every alternative (environment, containers, property wrappers)
exists because `init` injection becomes tedious at some scale. Start with
`init`; graduate only when the pain is real.

```swift
let repo = ProductRepository(api: ProductAPI(session: .shared),
                             cache: SwiftDataCache(context: modelContext))
let vm = ProductListViewModel(useCase: FetchProductsUseCase(repo: repo))
```

Every dependency is visible, every substitution explicit, nothing magic.

## SwiftUI's built-in DI: `@Environment`

SwiftUI already ships a DI system. Values put in the environment are
available to every descendant without prop-drilling.

```swift
private struct AuthSessionKey: EnvironmentKey {
    static let defaultValue: AuthSession = .unauthenticated
}
extension EnvironmentValues {
    var authSession: AuthSession {
        get { self[AuthSessionKey.self] }
        set { self[AuthSessionKey.self] = newValue }
    }
}

RootView().environment(\.authSession, session)  // inject at root
// @Environment(\.authSession) var session       // use anywhere below
```

Use this for truly ambient values (theme, locale, authenticated user). Do
*not* use it to sneak repositories into VMs — that's what `init` is for.
Environment values flow into previews, making them a great fixture mechanism.

## The Composition Root

*One* place at app launch builds the real object graph — typically `@main`
or an `AppDependencies` factory it owns.

```swift
@main
struct ShopApp: App {
    private let deps = AppDependencies.live
    var body: some Scene {
        WindowGroup {
            RootView()
                .environment(\.authSession, deps.authSession)
                .environment(deps.router)
        }
    }
}

struct AppDependencies {
    let authSession: AuthSession
    let productRepo: ProductRepositoryProtocol
    let orderRepo: OrderRepositoryProtocol
    let router: AppRouter
    static var live: AppDependencies { /* wire real implementations */ }
    static var preview: AppDependencies { /* wire in-memory fakes */ }
}
```

Two factories — `live` and `preview` — cover production and SwiftUI
previews with no conditional code. Tests build their own graphs directly.

Stay with manual init injection while the composition root is under ~150

## Coordinators — owning the "where next" decision

A Coordinator is an object whose single job is navigation. Views emit
intents ("user tapped Product X"); the coordinator decides whether that
opens a detail screen, a paywall, or a login sheet. In SwiftUI it becomes
a lightweight `@Observable` wrapper around `NavigationStack`'s path.

```swift
@Observable
final class AppRouter {
    var path = NavigationPath()
    var sheet: Sheet?

    enum Sheet: Identifiable, Hashable {
        case login, paywall, productDetail(Product.ID)
        var id: Self { self }
    }

    func open(_ product: Product) { path.append(ProductRoute.detail(product.id)) }
    func openCart() { path.append(CartRoute()) }
    func requireAuth() { sheet = .login }
    func reset() { path = NavigationPath() }
}

struct RootView: View {
    @Environment(AppRouter.self) private var router
    var body: some View {
        @Bindable var router = router
        NavigationStack(path: $router.path) {
            HomeView()
                .navigationDestination(for: ProductRoute.self) { route in
                    switch route {
                    case .detail(let id): ProductDetailView(id: id)
                    }
                }
                .sheet(item: $router.sheet) { sheet in
                    switch sheet {
                    case .login: LoginView()
                    case .paywall: PaywallView()
                    case .productDetail(let id): ProductDetailView(id: id)
                    }
                }
        }
    }
}
```

The router lives in the environment; VMs call `router.open(...)` without
knowing what a `NavigationStack` is. Deep links mutate `router.path` in
one line. Large apps subdivide into `CheckoutRouter` / `ProfileRouter` as
children of `AppRouter` so each feature owns its own navigation surface.

## An aside on TCA

[The Composable Architecture](https://github.com/pointfreeco/swift-composable-architecture)
takes a different path: state is a struct, actions are an enum, a reducer
maps `(state, action) → effects`, and dependencies flow through
`DependencyValues`. Over MVVM + Coordinators it adds exhaustive testing
(every state change and effect must be accounted for), navigation as
serializable state (`@Presents`, `Stack`, `Alert`), and time-travel
debugging via the TCA devtools.

The cost is real: boilerplate, a learning curve, and a strong opinion
about how logic is written. Reach for TCA when your app is heavily state-
driven (dashboards, editors, multi-step wizards) or needs bulletproof test
coverage. Start with [Point-Free's videos](https://www.pointfree.co).

Cheat sheet: stay manual until it hurts; reach for **Factory** first
(Swift-native ergonomics); pick **Swinject** for assembly-based registration
or mixed Obj-C; choose **Needle** when build-time graph correctness matters
more than onboarding speed.

## Key takeaways

- `init` injection is the default; alternatives earn their place by removing pain.
- `@Environment` is DI — use it for ambient values, not repositories.
- A Composition Root with `live` / `preview` factories beats global singletons.
- Routers turn navigation into testable state, not closures in buttons.

## Practice exercises

1. Build an `@Observable` `AppRouter` that owns a `NavigationPath` and a
   sheet enum. Inject it via `@Environment` and refactor one existing view
   to navigate through the router instead of `NavigationLink`.
2. Starting from manual init injection, introduce Factory (or Swinject) in
   a single feature. Measure the diff — is the composition root clearer
   or just differently verbose? Keep whichever is easier to onboard.
