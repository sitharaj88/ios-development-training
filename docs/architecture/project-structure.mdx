Folder layout is one of the few decisions you'll regret most if made carelessly.
This document shows the layout this curriculum uses, and the reasoning behind it.

Group **by feature**, not by file type. `ProductListView`, `ProductListViewModel`,
and `ProductRow` belong together. Splitting them across `Views/`, `ViewModels/`,
and `Components/` folders forces context switches every time you work on a feature.

## Modular layout (large apps)

For apps over ~30k lines or with multiple platforms (iOS + Watch + Mac), split
into Swift Packages:

```
MyApp/
в”њв”Ђв”Ђ MyApp.xcodeproj
в”њв”Ђв”Ђ App/                              # thin shell target
в””в”Ђв”Ђ Packages/
    в”њв”Ђв”Ђ Domain/
    в”‚   в””в”Ђв”Ђ Package.swift             # zero non-stdlib deps
    в”њв”Ђв”Ђ Data/
    в”‚   в””в”Ђв”Ђ Package.swift             # depends on Domain
    в”њв”Ђв”Ђ DesignSystem/
    в”‚   в””в”Ђв”Ђ Package.swift             # buttons, colors, typography
    в”њв”Ђв”Ђ Features/
    в”‚   в”њв”Ђв”Ђ ProductCatalog/
    в”‚   в”њв”Ђв”Ђ Cart/
    в”‚   в””в”Ђв”Ђ Checkout/
    в””в”Ђв”Ђ Networking/
```

Each package is independently testable, has its own `@testable import`, and
can be opened in its own Xcode window for fast iteration.

<Callout kind="tip" title="Use Tuist or XcodeGen for large monorepos">
Hand-edited `.xcodeproj` files merge poorly. Tuist and XcodeGen generate them
from declarative manifests вЂ” your team will thank you.
</Callout>

## The Composition Root

A single place where concrete dependencies are constructed and wired together,
typically at app launch.

```swift
@main
struct MyApp: App {
    let container = AppContainer()
    var body: some Scene {
        WindowGroup {
            RootView(viewModel: container.makeRootViewModel())
        }
    }
}

final class AppContainer {
    private lazy var api      = ProductAPI(session: .shared)
    private lazy var cache    = SwiftDataCache()
    private lazy var products: ProductRepository = ProductRepositoryImpl(api: api, cache: cache)

    func makeRootViewModel() -> RootViewModel {
        RootViewModel(useCase: FetchProductsUseCase(repo: products))
    }
}
```

This is the only place that knows the concrete `ProductRepositoryImpl` exists.
Every other file depends on the `ProductRepository` protocol.

## Test folder layout

Mirror the source layout:

```
Tests/
в”њв”Ђв”Ђ DomainTests/
в”‚   в””в”Ђв”Ђ UseCases/
в”‚       в””в”Ђв”Ђ CheckoutUseCaseTests.swift
в”њв”Ђв”Ђ DataTests/
в”‚   в””в”Ђв”Ђ Repositories/
в””в”Ђв”Ђ FeatureTests/
    в””в”Ђв”Ђ ProductList/
```

Each test file imports only what it needs. Rule of thumb: if a Domain test
needs `import UIKit`, the layering is wrong.
