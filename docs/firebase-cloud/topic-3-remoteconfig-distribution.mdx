## Firebase Remote Config

Remote Config is a server-side key/value store your app fetches at launch and
on demand. Think of it as feature flags + JSON-shaped config, with targeting
by audience, country, platform version, and random percentile.

```swift
import FirebaseRemoteConfig

@MainActor
final class FlagStore: ObservableObject {
    @Published private(set) var ready = false
    private let rc = RemoteConfig.remoteConfig()

    init() {
        let settings = RemoteConfigSettings()
        // Dev: fetch often. Prod: 12 hours is a reasonable default.
        settings.minimumFetchInterval = 3600
        rc.configSettings = settings
        rc.setDefaults(fromPlist: "RemoteConfigDefaults")
    }

    func start() async {
        do {
            _ = try await rc.fetchAndActivate()
            ready = true
        } catch {
            // Always fall back to baked-in defaults — never block the UI.
            ready = true
        }
    }

    var newOnboardingEnabled: Bool {
        rc["new_onboarding_enabled"].boolValue
    }
}
```

`RemoteConfigDefaults.plist` ships inside the app bundle with every key and a
safe default. If the network is down on first launch you still get a working
app — that's the whole point.

### Staged rollouts and kill switches

In the console set a **condition** like `Random percentile ≤ 10` and scope a
flag to it. Start at 1%, watch Crashlytics, ramp to 10%, 50%, 100% over a week.
Every big feature should also ship with a **kill switch** — a boolean flag that
forces the old code path. When a P0 bug lands on Friday evening you flip the
switch instead of cutting a hotfix.

<Callout kind="warning" title="Do not put user-facing strings in Remote Config for localization">
  It's tempting to A/B test copy by stuffing English strings into Remote Config
  keys. Don't. You break `NSLocalizedString`, lose plural/genitive handling,
  and every non-English user sees untranslated text. Use Remote Config to
  select a <em>variant id</em>, and keep the actual strings in `.xcstrings`
  where they can be localized properly.
</Callout>

## A/B Testing

Firebase A/B Testing sits on top of Remote Config + Analytics. You define an
experiment, pick an activation event, choose a primary metric (e.g. retention,
`purchase`), and Firebase randomly assigns users to variants and tracks
statistical significance.

Good experiment hygiene: one change per experiment, run for at least two
weekly cycles, pre-register your hypothesis, and ignore early "winners" before
the sample size is adequate.

**Recommendation:** use App Distribution for your internal ring (engineering,
design, QA) — fast iteration, no review friction. Promote release candidates
into TestFlight for wider beta. This two-ring model gives you speed *and* the
final Apple-verified sanity check before App Store submission.

CI wiring with Fastlane:

```ruby
lane :internal do
  build_app(scheme: "InboxApp", export_method: "ad-hoc")
  firebase_app_distribution(
    app: ENV["FIREBASE_APP_ID"],
    groups: "engineering,design",
    release_notes: last_git_commit[:message]
  )
end
```

### Native Apple alternatives

- **StoreKit 2 + iCloud KVS** for paywall flagging — if a flag only toggles
  subscription UI, you might not need a third-party service at all.
- **CloudKit public database** gives you a Firestore-like read-only config
  store scoped to your Apple Developer team, with no extra vendor to manage.
- **TestFlight + App Store Connect API** covers distribution and basic crash
  analytics for small teams that don't need Crashlytics' custom-keys power.

## Migrating away from Firebase

You'll know it's time when one of these hurts:

1. **Firestore query cost** dominates your Google Cloud bill — move hot paths
   to a server API backed by Postgres or DynamoDB.
2. **Security rules** become a second backend you can't test. Rewrite as a
   proper API with typed endpoints.
3. **Analytics sampling** at scale loses fidelity. Move to a data warehouse
   (BigQuery export is one click from the Firebase console).
4. **Vendor lock-in anxiety** from product. Mitigate by wrapping every Firebase
   call behind a repository protocol from day one, so swapping implementations
   is a week of work, not a quarter.

<Callout kind="info" title="Plan the exit on day one, don't execute it">
  Put a `UserRepository` protocol in front of Firebase Auth, a
  `MessagesRepository` in front of Firestore, and a `FileStore` in front of
  Storage. The abstraction costs almost nothing while you're in, and gives you
  a clean seam when you decide to leave.
</Callout>
