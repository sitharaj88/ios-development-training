## What this guide is

This is the same curriculum I teach to professional engineers transitioning into
the Apple ecosystem — refined into a self-paced reference site.
Every module is built around three principles:

1. **Explain the *why* first.** The Apple platform is opinionated. Understanding *why*
   `@Observable` replaced `ObservableObject`, *why* SwiftUI re-evaluates `body`, or
   *why* `actor` exists in the language shapes every architectural decision you'll make.
2. **Show production-grade code, not toys.** Every example assumes you're building
   something that real users will install — error handling, accessibility,
   localization, and testability are first-class concerns.
3. **End at the App Store.** The curriculum doesn't stop when the simulator runs.
   Code signing, TestFlight, App Store Connect, StoreKit 2, and ASO are part of
   the path.

<Callout kind="tip" title="If you came from another platform">
Coming from Android (Kotlin/Compose), React Native, Flutter, or web? You'll feel
at home faster than you expect — Swift's optionals map to Kotlin's nullables,
SwiftUI's declarative model echoes Compose and React, and Swift Concurrency
parallels Kotlin coroutines. We'll point out the analogies as we go.
</Callout>

## Old iOS vs. modern iOS

If you've seen iOS code from 2015–2019, the changes since are sweeping:

<ComparisonCard
  left={{

    variant: 'old',

      'Manual memory management mindset (retain cycles everywhere)',
      'Completion-handler callbacks for every async call',

      'Objective-C interop concerns front and center',
      'CocoaPods only; no first-party package manager',

  right={{

      'Compiler-enforced Sendable & data-race safety',

      'Pure Swift end-to-end (Obj-C only at framework boundaries)',
      'Swift Package Manager native in Xcode',
