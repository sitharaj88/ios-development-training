## Why this is still day-one material

- Most enterprise apps you will join have majority-UIKit screens.
- The AppKit/UIKit interop story (UIViewRepresentable,
  UIHostingController) requires you to understand the lifecycle on both
  sides.
- Debugging a misbehaving SwiftUI screen often means inspecting the hosting
  controller and the responder chain.
- The concepts — lifecycle hooks, delegation, responder chain — are the
  vocabulary Apple's own documentation uses everywhere.

## The UIViewController lifecycle

Each hook has a specific job. Putting code in the wrong one is the most
common UIKit bug.

<StepList
  steps={[
    { title: 'init(coder:) or init(nibName:bundle:)', description: 'The instance exists. Outlets are NOT connected yet. Use only for dependency injection.' },
    { title: 'loadView()', description: 'The system builds self.view. Override only if you create the view hierarchy entirely in code and want to skip the default UIView.' },
    { title: 'viewDidLoad()', description: 'Outlets are connected. The view is in memory but not yet on screen. One-time setup: register cells, configure subviews, attach observers.' },

    { title: 'viewDidAppear(_:)', description: 'Fully visible. Start animations, analytics screen events, request focus.' },
    { title: 'viewWillDisappear(_:) / viewDidDisappear(_:)', description: 'Pair with their appear counterparts. Tear down timers, save state, end editing.' },
    { title: 'deinit', description: 'Controller is being deallocated. Remove observers that used self if you did not use the block-based API.' },

```swift
final class ProductListVC: UIViewController {
    private let repository: ProductRepository
    private var refreshTimer: Timer?

    init(repository: ProductRepository) {
        self.repository = repository
        super.init(nibName: nil, bundle: nil)
    }
    required init?(coder: NSCoder) { fatalError("use init(repository:)") }

    override func viewDidLoad() {
        super.viewDidLoad()
        title = "Products"
        configureCollectionView()
    }

    override func viewWillAppear(_ animated: Bool) {
        super.viewWillAppear(animated)
        Task { await reload() }
    }

    override func viewDidAppear(_ animated: Bool) {
        super.viewDidAppear(animated)
        refreshTimer = Timer.scheduledTimer(withTimeInterval: 30, repeats: true) { [weak self] _ in
            Task { await self?.reload() }
        }
    }

    override func viewWillDisappear(_ animated: Bool) {
        super.viewWillDisappear(animated)
        refreshTimer?.invalidate()
        refreshTimer = nil
    }
}
```

<Callout kind="warning" title="Beware the massive view controller">
The oldest iOS antipattern: a view controller that owns networking, parsing,
persistence, analytics, and layout. The fix is not "smaller view controllers"
— it's <em>different objects</em>: a view model for state, a repository for
data, a router for navigation. Keep the VC to coordinating lifecycle and
binding outputs to views. MVVM, VIPER, and The Composable Architecture all
exist to solve this.
</Callout>

## UIView — frame vs bounds

Every view has two rectangles, and confusing them is how you spend an hour
debugging a layout that looks right in Interface Builder and wrong on device.

- **`frame`** — the view's rectangle in its **superview's** coordinate space.
  Changing `frame` moves or resizes the view.
- **`bounds`** — the view's rectangle in its **own** coordinate space. The
  origin is almost always `(0, 0)`; changing the origin scrolls the view's
  contents (this is how `UIScrollView` works).
- **Coordinate system** — origin top-left, y increases downward, units are
  points (not pixels). `UIScreen.main.scale` gives you the point-to-pixel
  ratio.

```swift
// Convert between coordinate spaces when you work across view hierarchies.
let pointInButton = button.convert(point, from: view)
let frameOnScreen = view.convert(view.bounds, to: nil)   // nil = window
```

Hit testing walks the view tree via `hitTest(_:with:)` + `point(inside:with:)`.
The deepest view whose `point(inside:...)` returns true wins the event. Set
`isUserInteractionEnabled = false` or override `hitTest` to customize.

## IBOutlets, IBActions & target-action

Even in code-only projects, **target-action** is how UIKit controls invoke
your methods. It's a runtime mechanism: the control holds a `(target, Selector)`
pair and sends a message for a given `UIControl.Event`.

```swift
// Code-only version — no Interface Builder required.
let button = UIButton(type: .system)
button.setTitle("Checkout", for: .normal)
button.addTarget(self, action: #selector(didTapCheckout), for: .touchUpInside)

@objc private func didTapCheckout() {
    Task { await checkout() }
}

// Modern closure-based alternative (iOS 14+):
button.addAction(UIAction { [weak self] _ in
    Task { await self?.checkout() }
}, for: .touchUpInside)
```

`@IBOutlet weak var label: UILabel!` is a property connected in Interface
Builder. `weak` is correct — the view hierarchy retains the label. The
implicit unwrap is pragmatic: after `viewDidLoad`, the outlet is never nil.

## The responder chain

Every `UIResponder` — `UIView`, `UIViewController`, `UIApplication` —
participates in a chain. When an event is not handled by the first
responder, UIKit walks up the chain looking for a method signature match:
view → superview → view controller → parent VC → window → application →
app delegate.

```swift
// Send an action up the chain without knowing who handles it.
UIApplication.shared.sendAction(#selector(UIResponder.copy(_:)),
                                to: nil, from: self, for: nil)
```

This is how the shake-to-undo gesture, keyboard shortcuts, and menu bar
items reach handlers without being wired to a specific object. `becomeFirstResponder()`
and `resignFirstResponder()` control keyboard focus.

## Gesture recognizers

A `UIGestureRecognizer` lives on a view and converts raw touches into
semantic events. Add one per gesture, configure with target-action or a
closure wrapper.

```swift
let tap  = UITapGestureRecognizer(target: self, action: #selector(handleTap))
let pan  = UIPanGestureRecognizer(target: self, action: #selector(handlePan))
let pinch = UIPinchGestureRecognizer(target: self, action: #selector(handlePinch))

view.addGestureRecognizer(tap)
view.addGestureRecognizer(pan)
view.addGestureRecognizer(pinch)

tap.require(toFail: pan)   // disambiguate overlapping gestures
```

Recognizers have a `state` machine (`.possible → .began → .changed → .ended`
or `.cancelled/.failed`). Read the state in your handler; don't just treat
`.ended` as the whole story for continuous gestures.

## Delegation — the protocol-based extension point

Delegation is UIKit's standard customization pattern: an object exposes a
protocol, you conform, you get called back. `UITableViewDelegate`,
`UIScrollViewDelegate`, `UICollectionViewDelegate`, `URLSessionDelegate`,
`CLLocationManagerDelegate` — the list is long.

```swift
// Your own delegate — use AnyObject so it can be weak.
protocol CheckoutViewDelegate: AnyObject {
    func checkoutView(_ view: CheckoutView, didConfirm order: Order)
    func checkoutViewDidCancel(_ view: CheckoutView)
}

final class CheckoutView: UIView {
    weak var delegate: CheckoutViewDelegate?
}
```

## Why SwiftUI still needs all of this

- **`UIViewRepresentable`** and **`UIViewControllerRepresentable`** are how
  you embed UIKit in SwiftUI. Their `Coordinator` is a delegate holder.
- **`UIHostingController`** is how SwiftUI views are embedded in UIKit nav
  stacks. Its `viewDidLoad`/`viewWillAppear` still drive `onAppear`/`task`.
- **`.introspect`** libraries read the underlying `UIKit` view to tweak
  behaviour SwiftUI doesn't expose.
- **Gesture recognizers** show up when SwiftUI's `DragGesture` doesn't
  compose the way you need.

Treat this topic as the bedrock. Every later module — SwiftUI, navigation,
networking — will reference it.

## Exercises

1. Build a `UIViewController` subclass that logs every lifecycle hook
   (`viewDidLoad`, `viewWillAppear`, `viewIsAppearing`, `viewDidAppear`,
   `viewWillDisappear`, `viewDidDisappear`, `deinit`). Push it, pop it,
   present it modally, dismiss it — confirm the order matches the table above.
2. Build a `CheckoutView` with a `CheckoutViewDelegate` protocol, two methods,
   and a weak delegate. Wire it to a host VC. Then add a second version of
   the same view that uses closures instead. Compare call-site readability
   and retain-cycle risk.
