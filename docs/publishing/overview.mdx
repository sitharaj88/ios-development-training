Sharing certificates via "export from Keychain" is a recipe for tears. `match`
stores them encrypted in a private Git repo and auto-installs on every dev
machine and CI runner.

## TestFlight in five steps

1. **Increment build number**, archive (Product → Archive).
2. **Distribute** → App Store Connect → upload.
3. App Store Connect processes the build (~10 min) and runs export compliance.
4. Add internal testers (no Apple review) or external (one-time review per major version).
5. Testers install via TestFlight app and feedback comes back through the same app.

## StoreKit 2 — IAPs that are pleasant

```swift
import StoreKit

let products = try await Product.products(for: ["pro_monthly", "pro_yearly"])

// Purchase
let result = try await products.first!.purchase()
switch result {
case .success(let verification):
    let transaction = try checkVerified(verification)
    await unlockPro(for: transaction)
    await transaction.finish()
case .userCancelled, .pending: break
@unknown default: break
}

// Listen for transaction updates anywhere in your app
Task.detached {
    for await update in Transaction.updates {
        // re-deliver content if needed
    }
}
```

The old StoreKit 1 receipt validation gymnastics are gone. Use server-side
verification with `Transaction.jsonRepresentation` and the App Store Server API
for revenue-critical paths.

## App Privacy is now part of submission

App Store Connect requires you to declare what data you collect, how it's used,
and which third-party SDKs collect data. Inaccurate disclosures = rejection
(or worse, removal). Audit your SDKs annually.

## Key takeaways

- **Code signing is process, not magic** — once you understand certs / profiles / capabilities, problems become diagnosable.
- **Use Fastlane match** in any team setting.
- **StoreKit 2 + server-side validation** is the modern IAP stack.
- **App Privacy disclosures are part of the binary** — keep them accurate.
- **Screenshots and the first 25 chars of your subtitle drive conversions** — invest there.
