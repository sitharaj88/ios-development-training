## Why signing exists at all

Every binary that runs on an iOS device is cryptographically signed by a
developer whose identity Apple has verified. That signature is what lets iOS
grant entitlements (Push, HealthKit, iCloud) and what lets the App Store attest
that the code you shipped is the code users downloaded. Without signing, the
iOS security model collapses.

The four artefacts you need to understand:

- **Apple Developer Program membership** — the legal contract.
- **Certificate** — the public/private keypair that proves *this machine signed the build*.
- **App ID** — the bundle-identifier-shaped entry that declares *which app* and *which capabilities*.
- **Provisioning profile** — the glue that says *this certificate can sign this App ID for these devices*.

## The signing trinity — in detail

### Certificate

A keypair stored in your Keychain. The private key signs; the public key is
uploaded to Apple. There are two categories:

- **Apple Development** — for running on real test devices.
- **Apple Distribution** — for App Store, Ad Hoc, and Enterprise builds.

Lose the private key and *no machine can produce an equivalent certificate* — Apple
cannot recover it. This is why teams use Fastlane `match` to store the keypair
encrypted in a private Git repo.

### App ID

Either **explicit** (`com.yourco.YourApp`) or **wildcard** (`com.yourco.*`). Only
explicit App IDs can use capabilities that need push tokens or app-specific
entitlements (Push Notifications, iCloud, Associated Domains, App Groups, Sign
in with Apple). Use explicit for anything real.

### Provisioning profile

A signed blob that says: *"These certificates are allowed to sign this App ID
and install on these devices, with these entitlements."* Three kinds:

- **Development** — lists allowed device UDIDs; expires in 12 months.
- **Ad Hoc** — lists allowed device UDIDs; for enterprise-distributed builds outside MDM.
- **App Store** — no device list; for upload to App Store Connect.

## App Store Connect API keys (.p8)

For any automation — Fastlane, Xcode Cloud, GitHub Actions — you need an API
key instead of your Apple ID password (which would require 2FA on every run).

```text
App Store Connect → Users and Access → Integrations → Keys
  → Generate API Key
  → Role: App Manager (or Developer for test-only access)
  → Download the .p8 ONCE — Apple never shows it again
```

You'll end up with three values: `KEY_ID`, `ISSUER_ID`, and the `.p8` file
contents. Store them in your CI's secret store, never in Git.

<Callout kind="warning" title="Never email .p12 files, never paste .p8 files in Slack">
A `.p12` export contains your distribution certificate's **private key**. Someone
with that file can ship malware under your company's name. A `.p8` App Store
Connect key can upload builds and manage users. Treat both like production
database passwords — encrypted Git repos (Fastlane match) or a real secrets
manager (1Password, AWS Secrets Manager), never chat or email.
</Callout>

## Archive → Distribute flow

```text
Product → Archive (requires a "Any iOS Device" destination, not a simulator)
  → Organizer opens
  → Distribute App
    → App Store Connect   (production upload)
    → TestFlight Internal Only (faster path for pre-release testing)
    → Ad Hoc              (signed IPA for specific UDIDs)
    → Enterprise          (in-house distribution)
    → Development         (local install on a registered device)
```

The Organizer stores every archive locally — keep at least one per released
version so you can symbolicate crash reports long after the fact.

## TestFlight: internal vs external

- **Internal testing** — up to 100 team members (users in App Store Connect). No
  Apple review. Builds appear in TestFlight within minutes of processing.
- **External testing** — up to 10,000 testers via email invite or a public link.
  First build of each version goes through **Beta App Review** (usually under 24
  hours, sometimes minutes). Subsequent builds of the same version ship without
  re-review.

Build expires after **90 days**. Plan your beta cadence accordingly.

## "Code signing failed" — the debugging flowchart

1. Does the bundle identifier in the target **exactly** match the App ID in the profile? (Case-sensitive.)
2. Is the profile **expired**? (Dev profiles: 12 months. Certificates: 12 months for dev, 12 months for distribution, 3 years for Apple Push.)
3. Does the profile include the certificate you're trying to sign with? (Re-download after regenerating a cert.)
4. Do the target's **entitlements** match the capabilities the App ID has enabled? (Adding Push in Xcode but not in the portal = failure.)
5. For CI: is the correct `.p12` + profile installed in the Keychain *before* `xcodebuild` runs?
6. For Automatic signing: is Xcode signed in with an Apple ID that belongs to the correct Team?

Ninety-nine percent of signing errors are one of these six.

## Exercises

1. **Manual-signing a fresh app.** Create a new app, disable Automatic signing, manually create a Development certificate and provisioning profile in the Developer portal, and install them. Archive and export an Ad Hoc IPA to your device.
2. **TestFlight CI upload.** Using an App Store Connect API key, upload a TestFlight build from the command line (`xcodebuild -exportArchive` + `xcrun altool --upload-app`) without opening Xcode Organizer.
