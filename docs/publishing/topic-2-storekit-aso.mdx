## Why StoreKit 2 exists

StoreKit 1 required base64-decoding ASN.1 receipts, round-tripping them to
Apple's `/verifyReceipt` endpoint, and parsing the response. Every app ended up
reinventing the same bugs. StoreKit 2 (iOS 15+) throws all that away: every
`Transaction` is a **signed JWS** you can verify locally, and the API is
async/await all the way down.

## A complete purchase flow

```swift
import StoreKit

@MainActor
final class Store: ObservableObject {
    @Published private(set) var products: [Product] = []
    @Published private(set) var purchasedIDs: Set<String> = []

    private var updatesTask: Task<Void, Never>?

    func bootstrap() async {
        updatesTask = Task.detached { [weak self] in
            for await result in Transaction.updates {
                await self?.handle(transaction: result)
            }
        }
        await loadProducts()
        await refreshEntitlements()
    }

    func loadProducts() async {
        do {
            products = try await Product.products(for: [
                "pro_monthly", "pro_yearly", "coins_500",
            ])
        } catch {
            print("Product load failed: \(error)")
        }
    }

    func purchase(_ product: Product) async throws {
        let result = try await product.purchase()
        switch result {
        case .success(let verification):
            let transaction = try verify(verification)
            await deliver(transaction)
            await transaction.finish()         // always finish, even after server grant
        case .userCancelled:
            break
        case .pending:
            break                              // SCA / Ask to Buy — wait for Transaction.updates
        @unknown default:
            break
        }
    }

    private func verify<T>(_ result: VerificationResult<T>) throws -> T {
        switch result {
        case .verified(let value):       return value
        case .unverified(_, let error):  throw error
        }
    }

    func refreshEntitlements() async {
        var owned: Set<String> = []
        for await result in Transaction.currentEntitlements {
            if case .verified(let tx) = result { owned.insert(tx.productID) }
        }
        purchasedIDs = owned
    }

    private func handle(transaction result: VerificationResult<Transaction>) async {
        if case .verified(let tx) = result {
            await deliver(tx)
            await tx.finish()
        }
    }

    private func deliver(_ transaction: Transaction) async {
        purchasedIDs.insert(transaction.productID)
    }
}
```

The `Transaction.updates` stream is the **only** reliable way to catch purchases
that happen outside your app (Ask to Buy approvals, Family Sharing shares,
subscription renewals). Start listening on launch, before restoring UI state.

## Server-side verification with the App Store Server API

Local JWS verification is fine for most apps, but revenue-critical features
should double-check on your backend. The App Store Server API accepts a
`transactionId` and returns a **signed JWS** you re-verify with Apple's public
keys.

```text
POST https://api.storekit.itunes.apple.com/inApps/v1/transactions/{transactionId}
Authorization: Bearer <JWT signed with your .p8 App Store Connect key>
```

The response's `signedTransactionInfo` is a JWS whose payload is the canonical
transaction record. Validate the JWS chain against Apple's root, then trust the
payload. Never trust a receipt forwarded from the client — signed JWS is the only
source of truth.

## StoreKit config files — testing without the sandbox

The sandbox store is slow, flaky, and clears entitlements unpredictably. Add a
**StoreKit Configuration File** to your target (File → New → StoreKit Configuration
File), list your products there, and enable it in the scheme's Run options.
Purchases happen instantly, subscriptions renew in seconds, and you can simulate
refunds, billing failures, and Ask to Buy.

```text
Scheme → Run → Options → StoreKit Configuration → StoreKit.storekit
```

## Promotional offers

Configured in App Store Connect, served via a **signed offer** from your server
(you sign with a key Apple generates for offers specifically, not your API key).
Use them to win back lapsed subscribers — "20% off for 3 months if you come back
this week."

## Privacy Nutrition Labels and Privacy Manifests

Two separate things that are both mandatory as of 2024+:

- **Privacy Nutrition Labels** — declared in App Store Connect, shown on the product page. What data you collect, whether it's linked to the user, whether it's used for tracking.
- **Privacy Manifest (`PrivacyInfo.xcprivacy`)** — a plist shipped *inside the binary*. Declares:
  - `NSPrivacyAccessedAPITypes` — justifies every use of "required reason" APIs (UserDefaults, file timestamps, disk space, system boot time, active keyboard).
  - `NSPrivacyTrackingDomains` — domains your app contacts for tracking, blocked by iOS when ATT hasn't been authorized.
  - `NSPrivacyCollectedDataTypes` — the machine-readable mirror of the nutrition label.

Every third-party SDK on Apple's "commonly used SDKs" list must now ship its own
`PrivacyInfo.xcprivacy` + a signature. If a dependency doesn't, App Review
rejects your upload with a list of missing manifests.

<Callout kind="warning" title="Privacy Manifest drift is the new signing drift">
Every SDK update can add new required-reason APIs. Run `grep -r NSPrivacyAccessedAPITypes` after every dependency bump and update your aggregated app-level manifest. Treat it as part of the release checklist.
</Callout>

## App Store Optimization — what actually moves the needle

- **App name (30 chars)** + **subtitle (30 chars)** are the highest-weighted ranking signals.
- **Keywords field (100 chars)** — comma-separated, not visible to users. Don't repeat words from the name/subtitle; those are already indexed.
- **Screenshots** — first three are visible without scrolling. Lead with the hero feature, not your onboarding screen.
- **App Preview video** — autoplays silently on the product page. 15–30 seconds, no voice-over, captions baked in.
- **Localization** — per-locale name + screenshots + keywords. A Japanese localization alone often doubles downloads in Japan.
- **Product Page Optimization** — A/B test up to 3 variants of screenshots / icon / preview against the default. Requires 90% traffic minimum per variant to reach significance.
- **Reviews and ratings** — prompt via `SKStoreReviewController.requestReview(in:)` after a success moment (completing a workout, saving a document), never on first launch.
