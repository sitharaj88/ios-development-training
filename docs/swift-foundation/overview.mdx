## Why Swift, and why so much detail up front

Swift was designed to be safe (the type system catches whole classes of bugs at
compile time), fast (no garbage collector — ARC instead), and expressive (value
types, generics, protocols, pattern matching). Most of what makes Swift "feel
right" once you're fluent is invisible if you skim the basics. So we don't.

<Callout kind="tip" title="Already know another typed language?">
If you know Kotlin, TypeScript, or Rust, much of Swift will feel familiar.
We'll move briskly through syntax and spend most of our time on
the *Swift-specific* mental models: value vs reference semantics, optionals,
protocol-oriented programming, and Swift Concurrency.
</Callout>

## Value types vs reference types — the foundational mental model

This single distinction shapes how you think about every Swift program.

```swift
struct Point { var x: Int; var y: Int }
class  Counter { var value = 0 }

var a = Point(x: 1, y: 2)
var b = a            // copy
b.x = 99
print(a.x)           // 1  ← unaffected

let c = Counter()
let d = c            // same instance
d.value = 42
print(c.value)       // 42 ← shared
```

Once this clicks, every other Swift design decision starts making sense.

## Optionals are not "nullable" — they are a type

```swift
var name: String  = "Sitharaj"      // can never be nil
var nick: String? = "S"             // Optional<String> — might be nil
nick = nil                          // ok

// Safe unwrap
if let nick { print("Hi \(nick)") }

// Early exit
guard let nick else { return }

// Default
let display = nick ?? "anonymous"
```

The compiler refuses to let you call `nick.uppercased()` directly — you must
unwrap first. Every `nil`-related crash you've seen in another language is
prevented at compile time here.

<Callout kind="warning" title="Avoid the bang">
Force-unwrapping (`nick!`) crashes if `nick` is `nil`. It exists so you can
escape the type system at boundaries (IBOutlets, fixtures), not as a shortcut.
The rule of thumb: if you're typing `!`, ask whether `if let` or `guard let`
would express your intent more honestly.
</Callout>

## Protocol-oriented programming

Swift's standard library is built on protocols, not class hierarchies.
Protocols + extensions + generics replace most of what other languages do
with abstract base classes.

```swift
protocol Identifiable {
    associatedtype ID: Hashable
    var id: ID { get }
}

extension Array where Element: Identifiable {
    func find(_ id: Element.ID) -> Element? {
        first { $0.id == id }
    }
}
```

You'll lean on this pattern constantly when designing repositories,
use cases, and view-model contracts in later modules.

## Swift Concurrency — the primitive every modern API uses

`async`/`await` is no longer optional knowledge — `URLSession`, `SwiftData`,
`HealthKit`, and most third-party libraries now expose async APIs.

```swift
func fetchUser(id: UUID) async throws -> User {
    let (data, _) = try await URLSession.shared.data(from: userURL(id))
    return try JSONDecoder().decode(User.self, from: data)
}

Task {
    do {
        let user = try await fetchUser(id: id)
        print(user.name)
    } catch {
        print("Failed: \(error)")
    }
}
```

We introduce it here at the syntax level. Module 12 covers actors,
`AsyncSequence`, `TaskGroup`, and the runtime model in depth.

## Key takeaways

- Swift is **safe by default** — optionals, immutability with `let`, and ARC are not optional concepts.
- **Structs are first** in modern Swift; reach for classes only when you need identity or shared mutable state.
- **Protocols + extensions + generics** replace inheritance for most abstraction needs.
- **`async`/`await`** is the new normal — every framework you'll touch exposes it.
