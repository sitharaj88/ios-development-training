## What makes SwiftUI different

Every SwiftUI view is a `struct` — a value, not a UIView. The framework calls
`body` to compute a description of the UI for the current state, diffs it against
the previous result, and only renders what changed.

<Diagram
  title="The SwiftUI render loop"
  caption="State change → tracked-read invalidation → body recompute → diff → render. Only views whose tracked state actually changed re-evaluate."
>
  <SwiftUIStateDiagram />
</Diagram>

This is the same model as React or Jetpack Compose, with one critical advantage:
SwiftUI tracks dependencies *automatically* via property wrappers, not by
manually-listed dependency arrays.

## State property wrappers — choose correctly

<ComparisonCard
  left={{

```swift
@Observable
final class Cart {
    var items: [CartItem] = []
    var total: Decimal { items.reduce(0) { $0 + $1.price } }
}

struct CartView: View {
    @Bindable var cart: Cart   // two-way bindings
    var body: some View {
        List(cart.items) { item in
            Text(item.name)
        }
        Text(cart.total, format: .currency(code: "USD"))
    }
}
```

## Modern navigation

`NavigationView` is deprecated. Use `NavigationStack` (iOS 16+) for a single-column
experience and `NavigationSplitView` for iPad/Mac multi-column UI.

```swift
NavigationStack(path: $path) {
    ProductList()
        .navigationDestination(for: Product.self) { product in
            ProductDetail(product: product)
        }
        .navigationDestination(for: CartRoute.self) { _ in
            CartView(cart: cart)
        }
}
```

Push `path.append(product)` from anywhere in the tree. The state is the source
of truth; the UI follows.

## Animations are state-driven

```swift
@State private var expanded = false

Image(systemName: "chevron.right")
    .rotationEffect(.degrees(expanded ? 90 : 0))
    .animation(.spring(duration: 0.3), value: expanded)
```

The view describes the *what*; SwiftUI interpolates the *how*. We cover
transitions, `matchedGeometryEffect`, and the Canvas API for custom drawing.

## When to drop down to UIKit

<Callout kind="info" title="UIKit interop is a feature, not a workaround">
SwiftUI provides `UIViewRepresentable` and `UIViewControllerRepresentable` for
this exact purpose. Use them for: `MFMessageComposeViewController`, advanced
camera UI, `WKWebView`, complex `UICollectionView` layouts, or vendor SDKs
that ship UIKit-only.
</Callout>

```swift
struct WebView: UIViewRepresentable {
    let url: URL
    func makeUIView(context: Context) -> WKWebView { WKWebView() }
    func updateUIView(_ webView: WKWebView, context: Context) {
        webView.load(URLRequest(url: url))
    }
}
```

## Key takeaways

- SwiftUI views are **values that describe state-to-UI**, not heavyweight UIView instances.
- Use **`@State` for local, `@Observable` for shared** — that covers 95% of cases.
- **`NavigationStack` with a typed path** is the modern navigation API.
- **Animations are declarative** — animate values, not properties.
- **UIKit interop is a tool**, not a fallback. Use it whenever it's the cleanest answer.
