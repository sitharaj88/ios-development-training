## Views are values

Every view conforms to `View` and is a `struct` — a value type. The framework
reads your `body` to get a *description* of what should be on screen, diffs it
against the previous description, and renders the delta.

```swift
struct Greeting: View {
    let name: String
    var body: some View {
        Text("Hello, \(name)")
            .font(.title)
    }
}
```

The `some View` return type is an opaque type — it says "I return one specific
type that conforms to View, and the compiler knows what it is." This matters:
SwiftUI needs the concrete type to diff efficiently. Returning `any View`
(existential) defeats the optimiser and costs you performance.

<Callout kind="tip" title="Extracting subviews is free">
Pulling a complex `body` into a smaller `struct MyRow: View` doesn't cost
anything at runtime. SwiftUI's diffing operates on the type tree, so smaller,
named views actually help — they give SwiftUI more granular identity.
</Callout>

## Modifier order matters

A modifier returns a *new view* that wraps the previous one. That means the
order you chain them changes the result.

```swift
// A — coloured pill with inner padding
Text("Save")
    .padding(.horizontal, 16)
    .padding(.vertical, 8)
    .background(.blue, in: .capsule)
    .foregroundStyle(.white)

// B — text tightly wrapped in blue, floating inside whitespace
Text("Save")
    .background(.blue, in: .capsule)
    .padding(.horizontal, 16)
    .padding(.vertical, 8)
```

Read modifier chains bottom-up: "a capsule-backgrounded Text, padded on all
sides." Each modifier is a function that wraps its input.

## Composition with ViewBuilder

`@ViewBuilder` is the result-builder behind every container. It lets you write
imperative-looking code that constructs a static view tree.

```swift
VStack(alignment: .leading, spacing: 8) {
    Text("Title").font(.headline)
    if showSubtitle {
        Text("Subtitle").foregroundStyle(.secondary)
    }
    ForEach(tags, id: \.self) { Text($0) }
}
```

The `if` and `ForEach` are legal because `VStack`'s initialiser takes a
`@ViewBuilder` closure. When you need to return heterogeneous views from your
own helper, annotate with `@ViewBuilder`:

```swift
@ViewBuilder
func badge(for state: Status) -> some View {
    switch state {
    case .ok:    Image(systemName: "checkmark.circle.fill").foregroundStyle(.green)
    case .warn:  Image(systemName: "exclamationmark.triangle.fill").foregroundStyle(.orange)
    case .error: Text("Failed").foregroundStyle(.red)
    }
}
```

`Group` exists for the rare case where you need to apply a modifier to a block
of siblings without adding a stack.

## Stacks, spacing, alignment

The three stacks cover 90% of layouts:

```swift
VStack(alignment: .leading, spacing: 12) { ... }  // top-to-bottom
HStack(alignment: .firstTextBaseline, spacing: 8) { ... }  // left-to-right
ZStack(alignment: .bottomTrailing) { ... }  // depth-ordered
```

`Spacer()` is a flexible view that pushes siblings apart. `Divider()` draws a
hairline separator. Both adapt to the stack axis.

For long lists or horizontal scrollers, swap to the lazy variants:

```swift
ScrollView {
    LazyVStack(spacing: 8) {
        ForEach(photos) { PhotoCell(photo: $0) }
    }
}
```

`LazyVStack` only instantiates rows as they approach the viewport. Use it for
anything over ~30 items, or when each row is expensive to build.

### Grid and LazyVGrid

`Grid` (iOS 16+) is a proper 2D grid with column alignment — rows are laid out
together:

```swift
Grid(alignment: .leading, horizontalSpacing: 16, verticalSpacing: 8) {
    GridRow { Text("Name");  Text(user.name) }
    GridRow { Text("Email"); Text(user.email) }
}
```

`LazyVGrid` takes a `[GridItem]` and lays out items flow-style — use it for
photo grids, not form layouts:

```swift
let columns = [GridItem(.adaptive(minimum: 100), spacing: 8)]
LazyVGrid(columns: columns, spacing: 8) {
    ForEach(photos) { PhotoThumb(photo: $0) }
}
```

## Sizing — frame, padding, layoutPriority

SwiftUI's layout protocol is a two-phase negotiation: parent proposes a size,
child returns its preferred size, parent places it. Your modifiers shape each
step of that conversation.

```swift
Text(longString)
    .frame(maxWidth: .infinity, alignment: .leading)   // propose full width
    .padding()                                          // inset content
    .layoutPriority(1)                                  // win the tie-break
    .fixedSize(horizontal: false, vertical: true)       // never truncate height
```

- `frame(...)` sets proposed dimensions. `maxWidth: .infinity` means "take all
  you're offered."
- `padding(...)` adds outside space — it's a size modifier, not a margin hint.
- `layoutPriority(_:)` breaks ties when siblings compete for space in an HStack.
- `fixedSize(...)` tells a view "use your ideal size on this axis, ignore the
  proposal" — the fix for multi-line text that truncates unexpectedly.

## GeometryReader — the reluctant tool

`GeometryReader` gives you the parent's proposed size so you can do manual
math. It also fills all available space by default, which catches everyone out.

<Callout kind="warning" title="Don't reach for GeometryReader first">
If your instinct is "I need GeometryReader to center this," you almost
certainly don't. Try `.frame(maxWidth: .infinity)`, alignment guides,
`ViewThatFits`, or the custom `Layout` protocol first. `GeometryReader`
breaks intrinsic sizing and makes your view expand to fill its parent —
often causing cascading layout surprises.
</Callout>

Valid uses: reading size to drive a parallax effect, sizing a custom shape to
a percentage of available width, measuring text to decide a wrap strategy.

```swift
GeometryReader { proxy in
    Rectangle()
        .fill(.blue.gradient)
        .frame(width: proxy.size.width * 0.6)
}
.frame(height: 120)   // constrain it — otherwise it eats everything
```

## ViewThatFits

`ViewThatFits` tries each child in order and picks the first that fits the
proposed size. Brilliant for adaptive layouts without media queries:

```swift
ViewThatFits(in: .horizontal) {
    HStack { Label("Edit", systemImage: "pencil"); Label("Share", systemImage: "square.and.arrow.up") }
    VStack { Label("Edit", systemImage: "pencil"); Label("Share", systemImage: "square.and.arrow.up") }
    Menu("Actions") { Button("Edit") {}; Button("Share") {} }
}
```

Wide screens show the HStack. Medium shows the VStack. Narrow shows the Menu.
Zero conditionals.

## Custom Layout protocol (teaser)

When the built-ins don't fit, implement `Layout` directly. You get two methods:
`sizeThatFits(proposal:subviews:)` and `placeSubviews(in:proposal:subviews:)`.
Flow layouts, radial menus, masonry grids — all doable in ~40 lines.

```swift
struct FlowLayout: Layout {
    var spacing: CGFloat = 8
    func sizeThatFits(proposal: ProposedViewSize, subviews: Subviews, cache: inout ()) -> CGSize { /* ... */ }
    func placeSubviews(in bounds: CGRect, proposal: ProposedViewSize, subviews: Subviews, cache: inout ()) { /* ... */ }
}
```

We revisit `Layout` in the advanced module. For now, know it exists so you
don't hack a FlowLayout out of nested GeometryReaders.

## Key takeaways

- Views are **values describing UI for the current state**, not UIView instances.
- Modifier **order is function composition** — read chains bottom-up.
- Use **`VStack`/`HStack`/`ZStack` eagerly**, `LazyVStack` when rows are numerous or expensive.
- **`frame` + `layoutPriority` + `fixedSize`** solve 90% of "why is this truncating" bugs.
- **`GeometryReader` is a last resort** — `ViewThatFits` and `Layout` usually do what you want.
